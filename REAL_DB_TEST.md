# Real-Database Verification ‚Äî Prisma mode against PostgreSQL

**Question:** the persistence / relations / M2M / transaction behaviour was so
far proven against a *fake* Prisma client. Does it hold against a **real
`@prisma/client` and a real database**?

**Answer: yes ‚Äî every scenario passes against real PostgreSQL, including the
one the fake could not truly prove (concurrent ACID decrement). The fake's
behaviour is confirmed.**

## Environment used

| Item | Value |
|---|---|
| Database | **Real PostgreSQL 16.13** (local cluster, not SQLite) |
| Client | `@prisma/client` **5.22.0** (matches the bundled `prisma` CLI 5.22.0) |
| `DATABASE_URL` | `postgresql://zerotest:zerotest@127.0.0.1:5432/zeroapi_test` |
| Schema | generated by `generatePrismaSchema(spec)` ‚Üí `realdb/prisma/schema.prisma` |
| Tables | created with real `prisma db push` (10 tables) + `prisma generate` |
| Runtime | `createRuntime(spec, { prisma: new PrismaClient() })`, HTTP via `app.request` |

So this is **Postgres, not SQLite** ‚Äî no SQLite-vs-Postgres caveats apply, and
the concurrency proof uses Postgres **row-level locking**.

Reproduce:

```bash
npm install --no-save @prisma/client@5.22.0
pg_ctlcluster 16 main start
export DATABASE_URL='postgresql://zerotest:zerotest@127.0.0.1:5432/zeroapi_test'
npx tsx realdb/gen-schema.ts
npx prisma db push --schema=realdb/prisma/schema.prisma
npx tsx realdb/run.ts          # the 8 scenarios below
npx tsx realdb/concurrency.ts  # the hardened concurrency proof
```

### Persistence (the real proof)
`POST /todos` returns 201 and the row is physically in the `Todo` table
(verified with `prisma.todo.findUnique`). A **brand-new `PrismaClient` + new
`createRuntime`** (simulating a process restart) reads the same todo back ‚Äî
the data lives in Postgres, not in any process memory. This is the proof the
in-memory fake could not give.

### Relations ‚Äî nested includes
`GET /posts/:id?include=comment.author` returns the post with its **comments
array, each carrying its author object**, resolved by Postgres in a single
query (native Prisma `include: { comments: { include: { author: true } } }`).
Depth > 1 confirmed on a real engine.

### Many-to-many
- `?include=hashtag` returns the post with `postHashtags[].hashtag` populated
  (join walked natively).
- `?hashtag=<id>` filters posts to only those linked to that hashtag
  (`where: { postHashtags: { some: { hashtagId } } }` pushed to Postgres).

### Self many-to-many
The self-M2M schema (`Person` ‚Üî `Person` via `Follows`) ‚Äî previously P1012 ‚Äî
is now valid; `prisma db push` created the `Follows` table with two distinct
FK columns, persons are queryable, and a follow edge persists.

## Results ‚Äî `realdb/concurrency.ts` (deterministic ‚úÖ)

The critical test the fake could only approximate. Real concurrent HTTP
requests, real Postgres row locks:

```
‚úÖ repeat#1 (expect 1√ó201/9√ó409): stock=1 conc=10 ‚Üí 201=1 409=9 finalStock=0 purchases=1
‚úÖ repeat#2 ‚Ä¶                      stock=1 conc=10 ‚Üí 201=1 409=9 finalStock=0 purchases=1
‚úÖ repeat#3 ‚Ä¶                      stock=1 conc=10 ‚Üí 201=1 409=9 finalStock=0 purchases=1
‚úÖ repeat#4 ‚Ä¶                      stock=1 conc=10 ‚Üí 201=1 409=9 finalStock=0 purchases=1
‚úÖ repeat#5 ‚Ä¶                      stock=1 conc=10 ‚Üí 201=1 409=9 finalStock=0 purchases=1
‚úÖ multi-winner (expect 3√ó201/17√ó409): stock=3 conc=20 ‚Üí 201=3 409=17 finalStock=0 purchases=3
‚úÖ high (expect 5√ó201/45√ó409):         stock=5 conc=50 ‚Üí 201=5 409=45 finalStock=0 purchases=5
```

**Exactly `stock` winners every single time**, across 5 repeats and higher
concurrencies (3/20, 5/50). The guarded
`updateMany({ where: { id, stock: { gte: amount } }, data: { stock: { decrement } } })`
inside `prisma.$transaction(...)` lets Postgres enforce guard + write atomically
under a row lock: the second waiter re-evaluates the guard against the freshly
committed `stock` and is rejected ‚Üí 409. Stock never goes negative; the count
of committed purchases equals the number of 201s. **This is the genuine
multi-process-grade proof the fake could only emulate.**

## Does the fake match real Prisma?

**Yes, for every behaviour tested.** The runtime emitted exactly the calls the
fake assumed, and real Postgres reacted exactly as the fake did ‚Äî same status
codes, same nesting, same 1-winner concurrency outcome.

### Differences / honest notes

1. **Concurrency model ‚Äî real is *stronger* than the fake.** The fake
   *serialized all* `$transaction` calls globally to emulate isolation. Real
   Postgres only locks the **contended row**, so transactions on *different*
   products run truly in parallel while still giving exactly-one-winner per
   row. The fake was a conservative under-approximation; the real engine is
   better, and the guard logic is what makes both correct.
2. **Timestamps are runtime-controlled, and Prisma honours that.** The runtime
   sets `id` (UUID), `createdAt` and `updatedAt` (ISO strings) explicitly.
   Despite the schema's `@default(cuid())` / `@default(now())` / `@updatedAt`,
   the **runtime-supplied values win** ‚Äî verified: after a `PUT`, the DB
   `updatedAt` equals the value the runtime sent (the row's id is the UUID, not
   a cuid). No breakage; just worth knowing the schema attributes are overridden
   by the runtime.
3. **Nothing broke** that the fake had hidden. The `@updatedAt` + explicit-value
   combination I had flagged as a risk does **not** error on real Prisma.
4. **One local test artefact (not a product bug):** installing `@prisma/client`
   for this exercise flips the premise of the unit test
   *"tryAutoLoadPrismaApiKeyStore returns null when @prisma/client is not
   installed"* ‚Äî it asserts the absent-Prisma path, so it fails while the package
   is present. `@prisma/client` is intentionally **not** a dependency (it stays
   optional / auto-detected), so the committed package keeps all 1003 unit tests
   green; this failure exists only in this provisioned environment.

### SQLite vs Postgres
Not applicable ‚Äî a real **PostgreSQL** server was available and used, so the
results reflect production-grade row-level concurrency directly.

## Verdict

The Prisma-mode persistence, nested relations, many-to-many (including self-M2M
and filtering) and ACID transactions all behave on a **real PostgreSQL database**
exactly as proven against the fake ‚Äî and the concurrent-decrement guarantee,
the one thing a single-process fake could not truly establish, holds
deterministically under real database concurrency.
